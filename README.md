# TH_R2

Cloudflare R2 remote media storage integration for Magento 2 and Adobe Commerce.

This module is planned as a package-style Magento module. Package metadata stays
at the package root, while the Magento module source code lives in `src/`.

## Description

`TH_R2` stores and retrieves Magento media files from Cloudflare R2 by using the
S3-compatible R2 API through `aws/aws-sdk-php`.

The MVP focuses on product media first: product image upload, product image
retrieval, and migration of existing product media from local `pub/media` to
Cloudflare R2. The implementation direction is to integrate with Magento's
remote storage layer so the same storage driver can also support product
description media, short description media, and generated image editing/cache
files.

## Goals

- Store Magento media files in Cloudflare R2.
- Retrieve media files from Cloudflare R2 for storefront and admin usage.
- Support product images across Adobe Commerce product types.
- Support media used in product descriptions and short descriptions.
- Support generated image files produced by Magento image processing flows.
- Provide flexible configuration for R2 credentials, bucket, prefix, endpoint,
  and public media URL.
- Provide a command to migrate current media files to R2.
- Keep the storage implementation based on `aws/aws-sdk-php`.

## MVP Scope

The first implementation phase should include:

- Magento module `TH_R2`.
- Remote storage driver code, recommended as `cloudflare-r2`.
- Cloudflare R2 client configuration using:
  - Account ID.
  - Endpoint: `https://<ACCOUNT_ID>.r2.cloudflarestorage.com`.
  - Region: `auto`.
  - Bucket name.
  - Access key ID.
  - Secret access key.
  - Optional bucket prefix.
  - Optional public base URL or custom domain.
  - Optional path-style endpoint compatibility setting.
- Product image upload to R2.
- Product image retrieval from R2.
- Compatibility target for all Adobe Commerce product types:
  - Simple products.
  - Configurable products.
  - Grouped products.
  - Bundle products.
  - Virtual products.
  - Downloadable products.
- Migration command for copying current local media to R2.
- Connection test command for validating credentials and bucket access.

## Media Coverage

The module should be designed to support these media categories:

- Product base images.
- Product small images.
- Product thumbnails.
- Product gallery images.
- Media inserted into product descriptions.
- Media inserted into product short descriptions.
- Files generated by Magento image resizing and editing flows.
- Existing media under `pub/media` during migration.

## Out Of MVP

These features are intentionally not part of the first implementation:

- Direct browser-to-R2 uploads.
- Presigned upload URL generation.
- Private media authorization.
- Multi-bucket routing.
- CDN image transformation.
- Automatic deletion of local media after migration.
- R2 lifecycle rule management.
- R2 bucket creation from Magento.
- Advanced admin dashboard or usage analytics.

## Installation

Require the package in Magento:

```bash
composer require th/module-r2
```

Enable the module:

```bash
bin/magento module:enable TH_R2
bin/magento setup:upgrade
bin/magento cache:flush
```

For production mode, run dependency injection compilation and static content
deployment as required by the project deployment process.

## Architecture Direction

`TH_R2` should integrate with Magento's remote storage abstraction instead of
adding product-type-specific upload plugins.

Recommended approach:

- Register a `cloudflare-r2` driver in Magento's remote storage driver factory
  pool.
- Build an R2-compatible S3 client with `Aws\S3\S3Client`.
- Use `aws/aws-sdk-php` with the R2 endpoint and `auto` region.
- Reuse Magento remote storage behavior for media writes and reads.
- Keep product media compatibility at the filesystem/storage layer.
- Add a public media URL resolver only if the default remote object URL is not
  suitable for an R2 custom domain.

This approach should allow product image upload flows, WYSIWYG product media,
and generated image files to share the same remote storage backend.

## Configuration

The module should expose configuration values through Magento configuration or
deployment environment configuration.

Planned fields:

- Enable Cloudflare R2 storage.
- Account ID.
- Endpoint.
- Bucket.
- Region.
- Access key ID.
- Secret access key.
- Prefix.
- Public base URL.
- Path-style endpoint mode.
- Connection timeout.
- Retry count.

Recommended defaults:

- Endpoint: `https://<ACCOUNT_ID>.r2.cloudflarestorage.com`.
- Region: `auto`.
- Driver code: `cloudflare-r2`.
- Prefix: empty.
- Path-style endpoint mode: disabled unless required by compatibility testing.

Secrets must not be hardcoded. Access key ID and secret access key should be
stored through Magento encrypted configuration or deployment configuration.

Admin configuration is available at:

```text
Stores > Configuration > TH > Cloudflare R2
```

The admin configuration is used by `bin/magento th:r2:test-connection` and as a
fallback for the `cloudflare-r2` driver factory. For real Magento media upload
through the remote storage filesystem, configure Magento deployment config with
the `cloudflare-r2` driver:

```php
'remote_storage' => [
    'driver' => 'cloudflare-r2',
    'prefix' => '',
    'config' => [
        'account_id' => '<cloudflare_account_id>',
        'endpoint' => 'https://<cloudflare_account_id>.r2.cloudflarestorage.com',
        'bucket' => '<bucket>',
        'region' => 'auto',
        'credentials' => [
            'key' => '<access_key_id>',
            'secret' => '<secret_access_key>',
        ],
        'path-style' => '0',
    ],
],
```

If `endpoint` is omitted, `TH_R2` can build it from `account_id`.

## Public Media URL

For production storefront media URLs, use a Cloudflare R2 custom domain.

Cloudflare's `r2.dev` public URL is useful for development and testing, but it
should not be treated as the production media URL. A custom domain gives better
control over caching, access policies, WAF rules, and production traffic.

## Commands

### Test Connection

Planned command:

```bash
bin/magento th:r2:test-connection
```

Expected behavior:

- Validate required configuration.
- Create an R2 S3 client.
- Check bucket access.
- Optionally write, read, and delete a temporary test object.
- Return a clear success or failure message.

### Media Migration

Planned command:

```bash
bin/magento th:r2:media:sync
```

Expected behavior:

- Copy current media files from local `pub/media` to Cloudflare R2.
- Preserve relative paths.
- Skip files that already exist remotely.
- Report copied, skipped, and failed counts.
- Print failed file paths with error messages.
- Support a dry-run mode before uploading.

The current command wraps Magento's remote storage synchronizer. Dry-run, path
filtering, and force upload options are planned for a later phase.

## Migration Flow

Recommended rollout for existing stores:

1. Configure the R2 bucket and credentials.
2. Configure the R2 public custom domain.
3. Run `bin/magento th:r2:test-connection`.
4. Run `bin/magento th:r2:media:sync --dry-run`.
5. Run `bin/magento th:r2:media:sync`.
6. Enable the R2 remote storage driver.
7. Flush Magento cache.
8. Validate product media in admin and storefront.
9. Keep local media as rollback source until production validation is complete.

## Compatibility

Target compatibility:

- Magento Open Source 2.4.x.
- Adobe Commerce 2.4.x.
- PHP versions supported by the target Magento version.
- `aws/aws-sdk-php` version 3.
- Cloudflare R2 S3-compatible API.

The first implementation should be validated on the exact project Magento
version before publishing a strict compatibility matrix.

## Known Limitations For MVP

- Local media deletion is not automated.
- Private media access is not implemented.
- Direct browser uploads are not implemented.
- Production custom domain setup is handled in Cloudflare, not in Magento.
- Bucket lifecycle, retention, and object lock settings are managed outside this
  module.
